## 2. The 9 required answers
1. **Connector source — GitHub only (D-28-1).** GitHub REST branch-protection (`GET /repos/{owner}/{repo}/branches/{branch}/protection`). The model is already GitHub-shaped (`PROVIDERS=("github",)`, `ci_evidence.py:26`). No GitLab/Bitbucket this slice.
2. **Project-repo + credential binding — resolved from the project's OWN intake (B1 / D-28-11/3/9).** The connector does **NOT** accept a caller-supplied `repo_ref` — that would let an unrelated protected repo (e.g. `torvalds/linux`) satisfy gate #3 for the wrong project. It resolves `repo_ref`/`branch` from the project's declared **`existing_assets_and_repositories`** category (`app/intake/categories.py:52`; `IntakeCategoryRepository.get_category`, `app/repositories/intake_categories.py:35-47`) and the credential **source** from **`secrets_and_credentials_manifest`** (`categories.py:56`, reference-only `{manager, reference_name}`, `categories.py:79`). The actual token **value** resolves from the operator env `GITHUB_CONNECTOR_TOKEN` (**fail-closed empty**, LLM-key pattern, `CLAUDE.md` `app/llm/`); per-tenant value resolution is **deferred (D-28-9)**. No raw token in DB/audit/params; no caller `repo_ref`. Missing/undeclared repo ⇒ fail-closed (no fetch).
3. **Broker path — read-only tool, decision-only broker, SAFE params (D-28-2 / B2-D-28-12).** Add ONE registry contract `source_control.read_branch_protection` → a **new low-privilege read action** `read_source_control_config` (A1+, `requires_approval=False`, non-§2.6) — a **read**, never a mutating action, so broker authority is **not broadened**. The connector calls `broker_call(...)` with **safe params only** — `{"provider":"github","branch":…,"repo_ref_present":true}` — and **never the raw `repo_ref`** (which `tool_calls.params` would persist and the secret denylist does **not** redact: `registry.py:15-23`, `app/repositories/tools.py:69-88`). It performs the real fetch **only** on an ALLOWED terminal (`ALLOWED_UNVERIFIED_IDENTITY` acceptable for a low-risk read). The broker stays **decision-only**; the connector is the executor (first real tool executor, broker-gated). `repo_ref` lives only inside connector execution + the snapshot row.
4. **Verified provenance — connector-only write path (D-28-4).** A new repo method `record_connector_verified_branch_protection` stamps `provenance="connector_verified"`, called **only** by the connector adapter after a verified fetch. `validate_new_snapshot` stays for the unverified path; a connector validator path accepts `connector_verified`. Migration `0027` does a **`CREATE OR REPLACE`** of the `0025` guard to **allow** `connector_verified` on INSERT while **preserving every other invariant** (repo_ref slug+token denylist, JSON-array, count-equality). **Honest caveat:** the DB widens the allowed value; **app-layer enforces *when*** it is used (the connector path is the sole writer) — the same app-enforced authenticity boundary as Slices 26/27 (`uaid_app` holds INSERT; raw SQL can't be distinguished). Stated plainly, never hidden.
5. **Gate #3 — repo-bound, latest-wins ladder (B4 / B1-cont / D-28-5/13):** (0) the project's `existing_assets_and_repositories` is undeclared/malformed ⇒ **`branch_protection_repo_unbound`** (**fail-closed — old snapshots cannot rescue this**); otherwise the gate resolves the **currently declared** `(provider, repo_ref, branch)` (shared `resolve_declared_repo`) and "latest" below is the **latest snapshot for that repo/branch** (`latest_branch_protection_for_repo`, **not** project-only). Then: (i) no snapshot for the declared repo ⇒ `no_branch_protection_evidence`; (ii) latest-for-repo provenance ≠ `connector_verified` ⇒ `branch_protection_observed_unverified` (**older verified ignored**); (iii) latest verified but stale/null `observed_at` ⇒ `branch_protection_evidence_stale`; (iv) latest verified+fresh but predicate false ⇒ `branch_protection_insufficient`; (v) latest verified+fresh **AND** `protection_enabled` **AND** `required_pull_request_reviews` **AND** `required_status_check_count >= 1` ⇒ **`passed`**. Freshness = `observed_at` within `CI_EVIDENCE_MAX_AGE` (24h, D-28-6). Counts are **context only**; the report carries `branch_protection_repo_bound` (bool), never raw `repo_ref`. Gate #3 passing adds **one** gate — `passed_gate_count` is **2 only when readiness is also `R5`**, **1** otherwise (B5).
6. **Tenant isolation.** `branch_protection_snapshots` is already tenant-owned RLS (`0025`); the connector writes via the tenant-scoped repo inside `tenant_scope`. `repo_ref` and the credential reference are **tenant/project-scoped** (from that tenant's intake); the connector never reads another tenant's repo, and the principal/repo never crosses tenants. Audit is tenant-GUC-derived.
7. **Failure behavior — fail-closed; `200`-only verified write (B3 / D-28-8).** Only a clean GitHub `200` (protection on) writes a `connector_verified` snapshot. **`404`** ("branch not protected" / missing branch / no access — these are **indistinguishable** on the protection endpoint), **`403`** (insufficient token scope), any non-200 / timeout / malformed response, missing/empty token, broker non-ALLOW, or undeclared project repo ⇒ **no verified write** (audited failure, safe metadata only). **No "verified-false" snapshot is ever written** (recording "protection is OFF" on a 404 would be a false claim). Insufficiency (protection on but no PR-reviews / 0 required checks) is captured by the **gate ladder** (item 5-iv), not a false snapshot. Min token permission: **repo-admin read** (branch-protection read requires admin). Stale evidence ⇒ gate not-pass. No partial/optimistic writes.
8. **Tests.** Pure (validator accepts connector tier on the connector path, rejects it on the caller path; gate #3 PASS/fail truth table incl. freshness); repository (verified write stamps `connector_verified`, counts move, latest ordering); DB-guard (direct SQL: `connector_verified` now allowed; bad provider/repo_ref/token/array still rejected; append-only still holds); broker path (FakeSCMConnector only — no network; connector executes only on ALLOW; deny ⇒ no write); gate #3 PASS (verified+active+fresh ⇒ pass) and fail (unverified / insufficient / stale / absent ⇒ not pass; go-live stays false); **no-secret audit** (token/repo_ref/URL/check-names never in audit or `tool_calls` params). `make test` + fresh `make test-db` + alembic round-trip.
9. **Non-goals.** No go-live beyond gate-#3 evidence (12 gates remain; `can_go_live_autonomously` stays false). No generalized multi-provider connector framework (GitHub-only, minimal adapter). No broad write API (the connector is **admin/internal-path**; no new HTTP write endpoint — the read-only `GET …/ci_evidence` is unchanged — D-28-7). No bypass of broker/approval/RLS. No PR/review/test-result evidence (that is Slice 29, gate #4). No real network in tests (Fake connector only). No live provider calls exercised in CI.

## 3. Security boundaries
- **Repo binding (B1):** the connector evidences **only the project's own declared repo** (`existing_assets_and_repositories`), never a caller-supplied one — an unrelated protected repo cannot satisfy gate #3.
- **Param minimization (B2):** `repo_ref` never enters `broker_call` params / `tool_calls.params` / audit — only `provider`/`branch`/`repo_ref_present`.
- **Credential containment:** token lives only in process env at call time; never persisted, never audited, never in `tool_calls` params (redacted), never in the snapshot/`repo_ref` (the token-prefix denylist `TOKENISH_RE` already rejects token-looking `repo_ref`, `ci_evidence.py:35`).
- **Authority containment:** the new tool is a **read** action; it cannot deploy/merge/mutate. Broker stays decision-only; the connector executes only on an ALLOWED decision; the allowlist still gates per-agent.
- **Provenance integrity:** `connector_verified` is app-stamped only on the verified path; the DB guard widens the allowed set but cannot itself attest authenticity (documented caveat). Gate #3 trusts only the **latest, fresh, verified** snapshot.
- **Honesty:** observed ≠ asserted; a `200` verified snapshot with **insufficient** protection (no PR-reviews / 0 required checks) **fails** the gate ladder; a `404`/`403`/error ⇒ **no snapshot** (never a fabricated "verified-off"). The connector never fabricates a pass.

- That `connector_verified` is DB-attested authenticity — it is **app-enforced** (the connector path is the sole writer; DB widens the allowed value).
